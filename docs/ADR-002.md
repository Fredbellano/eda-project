# ADR-002 : Choix de l'architecture — DDD + Hexagonale + Event-Driven

---

## Statut

Accepté

---

## Contexte

TaskFlow est une application de gestion de tâches collaboratives. Le domaine métier (projets, tâches, membres, états) doit être clairement délimité et nommé de façon cohérente entre l'équipe et le code. L'application nécessite du temps réel : lorsqu'un utilisateur déplace une tâche, les autres participants doivent voir le changement immédiatement. Par ailleurs, le backend ne doit pas être couplé à un type de client spécifique (web, mobile, etc.) — il doit pouvoir diffuser des événements de façon durable sans se soucier de qui les consomme. Enfin, l'architecture doit pouvoir absorber des changements d'infrastructure (changer d'ORM, de système de notification, de transport) sans impacter la logique métier.

---

## Décision

Nous avons décidé de combiner trois approches complémentaires :

- **DDD (Domain-Driven Design)** : pour définir un langage ubiquitaire et délimiter clairement les bounded contexts métier (Task, Project, Member, etc.). Le DDD structure le nommage et les responsabilités.
- **Architecture Hexagonale (Ports & Adapters)** : pour isoler la logique métier dans un noyau indépendant de l'infrastructure. Le domaine expose des ports, et les adaptateurs (BDD, HTTP, WebSocket) s'y branchent sans contaminer le cœur.
- **Event-Driven** : pour découpler le backend des consommateurs. Le backend émet des événements métier (`task.moved`, `member.added`) via un broker externe durable (NATS JetStream, détaillé dans ADR-003), et n'importe quel client ou service peut s'y abonner. Cela rend le temps réel naturel et le backend agnostique du type de client.

---

## Alternatives envisagées

| Alternative                          | Raison du rejet |
| ------------------------------------ | --------------- |
| Architecture Hexagonale seule (sans Event-Driven) | L'architecture hexagonale isole bien la logique métier, mais repose sur un modèle requête/réponse synchrone. Pour du temps réel sur une application de tâches collaboratives, cela devient lourd : il faudrait du polling ou des mécanismes ad hoc. L'Event-Driven apporte le temps réel de façon naturelle. |
| Event-Driven seul (sans Hexagonale) | Meilleure option que l'hexagonale seule pour le temps réel, mais sans l'abstraction Ports/Adapters, la logique métier risque de se mélanger avec l'infrastructure (handlers qui contiennent du SQL, de la logique HTTP, etc.). L'architecture hexagonale garantit que le cœur métier reste stable : seuls les adaptateurs changent quand l'infrastructure évolue. |
| Architecture modulaire | Bonne séparation par feature/module (`tasks/`, `projects/`…), mais sans l'abstraction formelle Ports/Adapters, la logique métier peut encore se mélanger à l'infrastructure au sein d'un module. Ne fournit pas non plus de découplage événementiel natif pour le temps réel. |
| Architecture en couches (MVC) | Couplage fort entre les couches. Pas d'isolation claire du domaine, pas de découplage événementiel. Ne répond ni au besoin de temps réel ni à l'objectif DDD. |

---

## Conséquences

**Positives :**

- Le nommage métier est partagé entre l'équipe et le code grâce au DDD (ubiquitous language)
- La logique métier est isolée et testable unitairement sans infrastructure (ni BDD, ni HTTP, ni WebSocket)
- Le backend est agnostique du client : il diffuse des événements, peu importe que le consommateur soit un site web, une app mobile ou un autre service
- La diffusion événementielle durable réduit le risque de perte de messages et facilite le rejeu en cas d'incident
- Ajouter un nouveau canal de diffusion (Slack, SMS) = créer un adaptateur, brancher sur le bus — zéro modification du domaine
- Le temps réel est une conséquence naturelle de l'architecture, pas un ajout technique forcé

**Négatives / compromis :**

- La combinaison DDD + Hexagonale + Event-Driven introduit plus de couches et de fichiers qu'une architecture MVC classique — la courbe d'apprentissage est plus élevée
- Le traçage d'un flux de bout en bout est moins linéaire : un événement peut être consommé par plusieurs handlers, ce qui complique le debug
- L'ajout d'un broker externe augmente la charge d'exploitation (supervision, configuration, alerting)
- Pour un projet de petite taille, le ratio boilerplate/logique métier peut sembler déséquilibré au départ

**Points de vigilance :**

- Ne pas mettre de logique métier dans les adaptateurs (handlers, controllers, repositories) — elle doit rester dans le domaine
- Nommer les événements avec le format `entité.action` (`task.moved`, `member.added`) pour respecter l'ubiquitous language
- Garantir l'idempotence des consumers et des handlers pour respecter le modèle at-least-once
- Documenter les bounded contexts et leurs interactions pour que l'équipe partage la même vision du découpage métier
