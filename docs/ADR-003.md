# ADR-003 : Fiabilite de publication des domain events avec NATS JetStream

---

## Statut

Accepté

---

## Contexte

Le fonctionnement de TaskFlow repose sur les evenements metier (`task.created`, `task.moved`, `task.assigned`, `member.added`).
Une perte d'evenement peut entrainer une incoherence fonctionnelle visible (notification absente, vue temps reel incomplete, audit incomplet).
Le systeme doit conserver un decouplage strict : un use case metier ne doit pas dependre de la reussite d'un handler specifique.
Nous ne souhaitons pas utiliser le pattern outbox.
Nous avons besoin d'une solution immediate, durable et exploitable qui reduit fortement le risque de perte d'evenements.

---

## Décision

Nous avons décidé d'utiliser un broker externe NATS JetStream comme colonne vertebrale des domain events.

Concretement :
- Le use case applique d'abord la modification metier en base.
- L'evenement est publie ensuite dans un stream JetStream (sujet au format `entite.action`).
- Les consommateurs sont durables, avec `ack` explicite apres traitement reussi.
- Une erreur dans un handler ne fait pas echouer le use case deja valide.
- Le retry est gere par JetStream (`maxDeliver` + backoff), puis redirection vers un sujet de type dead-letter apres epuisement.
- Chaque evenement porte un `eventId` unique ; tous les handlers sont idempotents pour tolerer les rejoues et doublons.

Le niveau de garantie retenu est **at-least-once** avec persistance des messages cote broker.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Bus interne en memoire | Risque plus eleve de perte en cas de crash process et absence de persistance native. |
| Kafka | Plus complexe a operer pour le perimetre du projet (cluster, tuning, exploitation). |
| RabbitMQ | Solution viable, mais JetStream est retenu pour son integration simple pub/sub + stream durable dans un seul composant. |
| Outbox pattern transactionnel | Non retenu par choix de conception du projet. |

---

## Conséquences

**Positives :**
- Durabilite native des messages via JetStream (stockage disque).
- Retry, accusés de reception et dead-letter geres nativement.
- Decouplage fort entre producteurs metier et consommateurs.
- Rejeu possible des evenements pour rattrapage ou reconstruction.

**Négatives / compromis :**
- Ajout d'une brique d'infrastructure a deployer et monitorer.
- Complexite operationnelle plus elevee (sujets, consumers, retention, supervision).
- Sans outbox, une garantie mathematique de perte nulle n'est pas prouvable dans tous les cas extremes entre commit DB et publication.
- Le modele at-least-once impose la gestion explicite des doublons cote handlers.

**Points de vigilance :**
- Imposer l'idempotence de tous les handlers (notification, websocket, audit).
- Tracer systematiquement `eventId`, `eventName`, `aggregateId`, `occurredAt` dans les logs.
- Surveiller le lag des consumers, les redeliveries, et la dead-letter queue.
- Ajouter des tests d'integration couvrant ack, retry, rejeu et non-regression metier.
