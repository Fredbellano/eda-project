# ADR-004 : Versionnement API REST et coexistence V1/V2

---

## Statut

Accepté

---

## Contexte

TaskFlow doit pouvoir faire evoluer son API sans casser les clients existants.
Certaines evolutions fonctionnelles peuvent introduire des changements de contrat (payload, semantics, routes).
Le projet doit permettre la coexistence de deux versions d'API exposees en parallele.
Le domaine metier doit rester stable et ne pas etre duplique a chaque version.
Nous devons donc definir une strategie de versionnement fiable

---

## Décision

Nous avons décidé d'utiliser un versionnement par URL, avec coexistence explicite de `/api/v1` et `/api/v2`.

Concretement :
- Chaque version expose ses propres endpoints HTTP et ses propres DTO de presentation.
- La logique metier reste unique et centralisee dans la couche application/domaine.
- Les differences entre versions sont gerees dans les adaptateurs entrants (controllers, mappers, validateurs).
- Les changements incompatibles sont introduits uniquement dans `v2`; `v1` reste stable hors correctifs critiques.
- Les deux versions sont couvertes par des tests de contrat independants.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Versionnement par header (`X-API-Version`) | Moins visible et plus difficile a deboguer pour une equipe et des consommateurs multiples. |
| Versionnement par media type (`Accept: application/vnd.taskflow.v2+json`) | Plus strict REST, mais plus verbeux et plus complexe a maintenir pour ce projet. |
| Version unique avec migration forcee | Risque de casser les clients existants, ne permet pas de coexistence progressive. |

---

## Conséquences

**Positives :**
- Version visible et explicite dans les routes, facile a comprendre et a tracer.
- Coexistence v1/v2 possible sans reecriture du domaine.
- Evolution plus sure : les ruptures de contrat sont isolees dans `v2`.
- Migrations client progressives avec reduction du risque de regression.

**Négatives / compromis :**
- Duplication partielle de la couche presentation (controllers, DTO, mapping).
- Charge de maintenance accrue tant que deux versions coexistent.
- Matrice de tests plus large (v1 + v2) et effort CI supplementaire.

**Points de vigilance :**
- Interdire toute logique metier dans les controllers v1/v2 pour eviter les divergences fonctionnelles.
- Garder des tests de non-regression sur v1 tant qu'elle est exposee.
- Definir une date de retrait de v1 et la communiquer explicitement.
- Mesurer l'usage de chaque version (logs/metrics) pour piloter la deprecation.
