# ADR-001 : Choix de la Stack Technique

---

## Statut

Accepté

---

## Contexte

Le projet TaskFlow démarre et nécessite de définir une stack technique claire et cohérente avant le début du développement. L'équipe possède une expérience significative avec Vue.js et Nuxt, mais moins avec React. Le projet est de taille modeste, orienté DDD avec une architecture hexagonale et event-driven. Le cahier des charges impose l'utilisation de Docker pour le déploiement. Le budget est limité : il faut privilégier des outils gratuits et open source. Enfin, le projet étant hébergé sur GitHub dans un dépôt public, l'outillage CI doit rester simple et intégré.

---

## Décision

Nous avons décidé d'adopter la stack suivante :

- **Frontend / BFF** : Nuxt (Vue.js) avec la librairie Nuxt UI pour accélérer le développement de l'interface.
- **Backend** : Go, pour sa rapidité d'exécution et sa simplicité qui facilite une structuration propre en DDD.
- **Base de données** : PostgreSQL avec GORM comme ORM.
- **Messagerie événementielle** : NATS JetStream pour la publication durable et le rejeu des domain events.
- **CI** : GitHub Actions.
- **Déploiement** : Docker.

---

## Alternatives envisagées

| Alternative                    | Raison du rejet |
| ------------------------------ | --------------- |
| React / Next.js (Frontend) | L'équipe est plus à l'aise avec Vue.js. Choisir React imposerait une courbe d'apprentissage supplémentaire et ralentirait le démarrage du projet. |
| Node.js / NestJS (Backend) | Envisagé brièvement mais écarté : Go offre une meilleure performance native, ne nécessite pas d'environnement d'exécution spécifique et sa simplicité convient mieux à un projet de cette taille. |
| MySQL / SQLite (BDD) | MySQL et PostgreSQL sont tous deux open source, mais PostgreSQL offre une licence plus permissive (PostgreSQL License vs GPL), un meilleur support des types avancés (JSON natif, arrays) et une meilleure conformité au standard SQL. SQLite ne convient pas à un contexte multi-utilisateur. |
| RabbitMQ / Kafka (Broker d'événements) | Solutions viables, mais non retenues : Kafka est plus complexe à opérer pour le périmètre du projet, et RabbitMQ a été écarté au profit de NATS JetStream pour une intégration plus simple entre pub/sub et persistance. |
| GitLab CI / Jenkins (CI) | GitHub Actions est natif à GitHub, gratuit pour un projet public, et son approche par briques d'actions est plus simple et rapide à mettre en place qu'une CI configurée de zéro. Pas besoin d'outil externe. |

---

## Conséquences

**Positives :**
- L'équipe travaille dans un environnement connu (Nuxt/Vue), ce qui accélère le démarrage et réduit les risques de blocage
- Nuxt UI fournit des composants prêts à l'emploi, réduisant le temps de développement front-end
- Go produit un binaire compilé autonome, simplifiant le déploiement Docker (image légère, pas de runtime)
- PostgreSQL + GORM offrent un écosystème mature et bien documenté pour Go
- NATS JetStream apporte la persistance des événements, les retries et le rejeu des messages
- GitHub Actions s'intègre sans friction avec le dépôt existant, sans configuration d'infrastructure supplémentaire
- Docker garantit un environnement reproductible pour l'examinateur

**Négatives / compromis :**
- Go possède un écosystème web moins riche que Node.js ou Java (moins de librairies/frameworks HTTP haut niveau)
- GORM, bien que populaire, peut masquer la complexité des requêtes SQL et encourager des patterns non optimaux si mal utilisé
- Nuxt UI impose ses conventions de design, ce qui peut limiter la personnalisation visuelle
- L'introduction d'un broker externe ajoute une complexité d'exploitation (configuration, supervision, alerting)

**Points de vigilance :**
- Veiller à ne pas laisser GORM fuiter dans la couche domaine : l'ORM reste dans la couche infrastructure (cohérence avec l'architecture hexagonale)
- Configurer JetStream avec des durées de rétention, politiques d'ack et retries cohérentes avec les besoins métier
- Imposer l'idempotence des consommateurs d'événements pour gérer les doublons du modèle at-least-once
- Maintenir les GitHub Actions simples et lisibles — ne pas accumuler de workflows complexes inutilement
- S'assurer que les images Docker restent légères (multi-stage build pour Go, image node alpine pour Nuxt)
